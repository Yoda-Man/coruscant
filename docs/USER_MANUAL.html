PostgreSQL Multi-Query Tool  ·  User Manual

1. Introduction

The Problem

If you have used pgAdmin, the most widely used GUI tool for PostgreSQL, you have likely run into this: you write a script with several SELECT statements, press Execute, and only the last result appears. The earlier result sets are gone. This is a known limitation of pgAdmin's current stable release. A fix is in development as of late 2025, but it has not yet been released to users.

The impact is real. A migration script that validates data before and after the change requires two queries. A diagnostic script may need five. Forcing each into its own separate run breaks the logical flow of the work and makes it harder to compare results side by side.

The Solution

Coruscant was built specifically to address this gap. Every statement in your script runs independently and produces its own result tab. Ten SELECT statements produce ten tabs, each one sortable, filterable, and exportable. You can pin the ones you want to keep, rename them for clarity, and compare them without re-running anything.

What Coruscant Does

  • Write and run SQL — syntax-highlighted editor with multi-tab support.
  • View results — sortable, filterable tables with live row filtering.
  • Browse your schema — tables, views, columns, indexes, foreign keys, and functions with hover-to-view definitions.
  • Generate scripts — right-click any table for a ready-to-edit SELECT, UPDATE, or DELETE script.
  • Manage transactions — auto-commit or manual mode with explicit Commit and Rollback.
  • Cancel queries cleanly — sends pg_cancel_backend() to the server; the UI never freezes.
  • Parameterized queries — named placeholders with safe server-side substitution.
  • Export results — CSV or JSON with a single click.
  • Load and save SQL files — open any .sql script and save your work to disk.

4.4 Testing a Connection

Click Test Connection before clicking OK. Coruscant opens a short-lived throwaway connection and reports success or failure without affecting your current session.

4.5 Recent Connections

The top of the dialog shows a drop-down of up to 5 previously used connections. Select one to auto-fill all fields. Passwords are stored encoded (not plaintext) in the OS settings store.

4.6 Disconnecting

Click Disconnect. Any open manual transaction is abandoned by the server automatically. Coruscant logs the disconnect and saves window geometry cleanly.

5. Writing SQL

5.1 The SQL Editor

  • Syntax highlighting: keywords (blue), functions (yellow), strings (orange), numbers (green), comments (grey).
  • Multiple statements: separate with semicolons. Each runs independently and produces its own result tab.
  • Tab key: inserts spaces for indentation.

5.2 Multiple Editor Tabs

ActionMethod
New tab+ Tab button or Ctrl+T
Next tabCtrl+Tab
Previous tabCtrl+Shift+Tab
Close tabCtrl+W or the × on the tab
Reorder tabsDrag to new position
If you close the last tab, the editor is cleared rather than removed — there is always at least one tab open.

5.3 Opening a SQL Script from File

  1. Click Open SQL… in the toolbar.
  2. Select a .sql or .txt file.
  3. The contents are loaded into the current editor tab.
Tip: Press Ctrl+T first to open a new tab, then click Open SQL… to avoid overwriting your current work.

5.4 Saving Your SQL to a File

  1. Click Save SQL… in the toolbar.
  2. Choose a folder and enter a filename, then click Save.

5.5 Formatting SQL

Click Format SQL to auto-format: keywords uppercased, identifiers lowercased, indentation normalised. If text is selected, only that portion is formatted.

Requires sqlparse: pip install sqlparse

5.6 Clearing the Editor

Click Clear to empty the editor and remove all unpinned result tabs. Pinned tabs are preserved.

6. Running Queries

6.1 Executing the Full Script

Press F5 or click ▶ Execute. Each statement produces its own result tab:

SELECT * FROM users;          -- result tab "Query 1"
SELECT count(*) FROM orders;  -- result tab "Query 2"
UPDATE products SET active = true WHERE id = 5;  -- "Query 3"SQL

6.2 Running a Selection Only

Highlight any portion of SQL before pressing F5. Only the selected text is sent to the server.

6.3 Setting a Row Limit

SettingBehaviour
1000 (default)Fetch at most 1 000 rows per SELECT
0 (Unlimited)Fetch all rows — use with caution on large tables

When truncated, a yellow warning banner appears at the top of the result tab.

6.4 Cancelling a Running Query

Click ■ Cancel or press Escape. PostgreSQL receives a cancellation signal. The status bar shows "Query cancelled." No error dialog appears.

7. Working with Results

7.1 Result Tab Types

What ranTab shows
SELECT or RETURNINGSortable, filterable table of rows
INSERT, UPDATE, DELETE, DDLSuccess message with rows affected
EXPLAIN / EXPLAIN+Query plan as monospace text
A failed statementError tab with full message — no popup dialog

7.2 Filtering Rows

Type in the Filter box to hide non-matching rows instantly. Case-insensitive, searches all columns. Clear the box to show all rows again.

7.3 Sorting Columns

Click any column header to sort ascending. Click again for descending. Click a third time to remove the sort.

7.4 Copying Rows to the Clipboard

Select rows (hold Ctrl or Shift for multiple) then press Ctrl+C. Rows are copied as TSV with a header row. Paste directly into Excel or Google Sheets.

7.5 Pinning and Renaming Result Tabs

Pinning: right-click a result tab → Pin. The tab gains a 📌 prefix and survives the next Execute. Right-click → Unpin to restore normal behaviour.

Renaming: right-click → Rename…, or double-click the tab title.

7.6 NULL Values

Database NULL values display as grey italic NULL, visually distinct from the string "NULL". In CSV exports they become empty cells.

7.7 Exporting Results

Export CSV — UTF-8 file with header row. NULL → empty string.

Export JSON — array of objects. Dates → ISO 8601, decimals → floats, binary → hex strings.

Exports always use the full unfiltered result set, regardless of what the filter box shows.

8. Transaction Mode

Auto-commit is on by default: every statement commits immediately. To use manual transactions:

  1. Uncheck Auto-commit in the toolbar.
  2. Run statements — changes are not visible to others until you commit.
  3. Click Commit to persist or Rollback to discard.
-- 1. Uncheck Auto-commit
DELETE FROM orders WHERE created_at < '2020-01-01';
SELECT count(*) FROM orders;
-- Looks right? Click Commit. Wrong? Click Rollback.SQL
DDL is transactional in PostgreSQL. In manual mode you can roll back a CREATE TABLE just like a DELETE.

9.3 Inserting a SELECT Statement

Double-click a table or view to insert a quick SELECT * at the cursor:

SELECT * FROM "public"."users" LIMIT 100;SQL

Double-click a function to insert a call template:

SELECT "public"."get_active_users"();SQL

9.4 Refreshing the Schema

The tree loads automatically on connect. Click Refresh in the Schema Browser header after schema changes (CREATE TABLE, ALTER TABLE, etc.).

Tip: If the Schema Browser shows nothing, check that you connected to your application database, not the postgres system database.

10. Query History

The Query History panel (bottom of the left dock) stores the last 100 queries with timestamps and execution times.

  • Double-click any entry to load the full SQL into the active editor tab.
  • Hover over an entry to see the complete SQL in a tooltip.
  • Click Clear History to delete all entries.
  • History persists across sessions.
Consecutive identical queries are deduplicated — re-running the same SQL updates the timestamp instead of adding a duplicate.

11. Parameterized Queries

Use %(name)s placeholders in your SQL, then supply values in the Parameters panel:

SELECT * FROM orders
WHERE customer_id = %(customer_id)s
  AND status      = %(status)s
  AND total       > %(min_total)s;SQL

Click Parameters ▸ in the editor header to expand the panel. Add a row for each placeholder, enter names and values, then press F5. Values are substituted via cursor.mogrify() — SQL injection is structurally impossible.

ParameterValue
customer_id1042
statusshipped
min_total50.00
The same parameter dictionary applies to all statements in the editor.

14. Logging

Coruscant writes a structured diagnostic log on every run.

Log file location

PlatformPath
Windows%APPDATA%\Coruscant\logs\coruscant.log
macOS~/Library/Logs/Coruscant/coruscant.log
Linux~/.local/share/Coruscant/logs/coruscant.log

Rotates at 5 MB, keeps 3 backups (15 MB maximum on disk).

What is recorded

LevelExamples
INFO (default)App start (version, Python, Qt, OS), connect/disconnect, schema loaded, query completed, theme changed, clean shutdown
WARNINGResult set truncated by row limit; query cancelled by user
ERRORConnection failures, query errors, schema errors
DEBUGFull SQL text, per-statement row counts and timing, Qt internal messages

Enabling DEBUG mode

rem Windows
set CORUSCANT_LOG_LEVEL=DEBUG
python main.py

# macOS / Linux
CORUSCANT_LOG_LEVEL=DEBUG python main.py

Crash reports

If Coruscant encounters an unhandled error it will write a CRITICAL log entry with the full traceback and show a dialog with the log file path.

Reading the log

2026-04-02 10:11:09.381 | INFO | coruscant.core.database | Connected  host=…

Monitor live (PowerShell):

Get-Content "$env:APPDATA\Coruscant\logs\coruscant.log" -Wait -Tail 50

Filter for errors:

Select-String -Path "$env:APPDATA\Coruscant\logs\coruscant.log" -Pattern "ERROR|WARNING|CRITICAL"

16. Troubleshooting

Before troubleshooting any issue: check the log file first. It records every connection attempt, query, and error with timestamps. See Section 14 for the file path and filtering commands.

“Could not connect” error

  • Check that the PostgreSQL server is running.
  • Verify host, port, database name, and credentials.
  • Check your firewall — port 5432 must be reachable.
  • Use Test Connection in the dialog for a detailed error message.
  • Try changing SSL mode to require for remote servers.

Schema Browser is empty after connecting

  • Check that you connected to your application database, not postgres.
  • Click Refresh in the Schema Browser header.
  • Confirm your PostgreSQL user has SELECT privileges on information_schema.
  • Check the log — a successful load records Schema loaded  schemas=N  tables=N. If N=0, a permissions issue is likely.

Query seems to hang

  • Click ■ Cancel or press Escape.
  • Use Explain+ to check for sequential scans or lock waits.

Auto-commit toggle is greyed out

Disabled while a query is running. Cancel or wait for it to finish first.

Format SQL does nothing

Install sqlparse: pip install sqlparse>=0.4

Error tab instead of results

Click ⚠ Error to read the full message. Fix the SQL and press F5 again.

App crashed without explanation

Open the log file and scroll to the end. Look for a CRITICAL entry with a full traceback. A dialog would also have appeared showing the log path.

17. Security Guidance

Saved Passwords

PlatformLocation
WindowsHKEY_CURRENT_USER\Software\Coruscant
macOS~/Library/Preferences/Coruscant.plist
Linux~/.config/Coruscant/Coruscant.conf
Base64 is not encryption. Anyone with access to the settings store can decode the passwords. On shared machines use ~/.pgpass instead and enter passwords manually.

SSL Recommendations

EnvironmentRecommended mode
Local developmentdisable or prefer
Internal trusted networkprefer
Internet / cloud databaserequire or verify-full
Regulated / sensitive dataverify-full

Least-Privilege Database Users

  • Read-only analysis → SELECT privileges only.
  • Schema changes → add CREATE, ALTER as needed.
  • Avoid connecting as postgres (superuser) for routine work.

SQL Injection

Coruscant uses cursor.mogrify() for parameterized queries, making SQL injection structurally impossible through the Parameters panel. If you manually concatenate user-supplied strings into SQL those queries are not protected — always use the Parameters panel for variable values.
